# Rua

Rua is a statically-typed scripting language with Rust-style syntax that compiles to readable Lua 5.5 source code. It retains `struct`, `enum`, `trait`, generics, pattern matching, closures, and iterators, but does not implement ownership, borrow checking, or lifetimes.

```bash
cargo build --release -p ruac -p rua-lsp --features lsp
target/release/ruac build app.rua
```

`ruac build` also emits a versioned artifact metadata file: bundle output gets
`<file>.rua-map.json`, while `--emit modules` gets `rua-artifact.json`. These
files let a host such as `moon_rs` run precompiled Lua and still map errors
back to Rua source.

Output targets Lua 5.5. The standard runtime is consolidated in a single [rua_std.lua](crates/rua-resources/resources/std/rua_std.lua); bundle codegen emits at most one `require("rua_std")`, while modules codegen relies on Lua's `require` cache to share the same runtime package.

Rua can embed a raw Lua block when an API is not yet wrapped by a `.ruai`
declaration:

```rua
lua! {
    local value = { ready = true }
    print(value.ready)
}
```

The block is emitted into the surrounding Lua chunk without Rua type checking or
variable interpolation. Use `extern "lua"` when a typed Lua ABI is preferable.

## Complete example: Rua and Lua

A Rua file is itself an executable chunk with no special entry-point function. The source below covers top-level statements, structs, associated functions, methods, enums, pattern matching, `Result`, `?`, Vec, HashMap, closures, iterators, and standard formatting functions:

```rua
struct Product {
    sku: String,
    price: i64,
}

impl Product {
    fn new(sku: String, price: i64) -> Product {
        Product { sku, price }
    }

    fn label(&self) -> String {
        format("{}: {}", self.sku, self.price)
    }
}

enum Checkout {
    Paid(Product),
    Rejected(String),
}

fn positive_quantity(quantity: i64) -> Result<i64, String> {
    if quantity > 0 {
        Ok(quantity)
    } else {
        Err("quantity must be positive")
    }
}

fn checkout(product: Product, quantity: i64) -> Result<Checkout, String> {
    let count = positive_quantity(quantity)?;
    let total = product.price * count;
    if total <= 10000 {
        Ok(Checkout::Paid(product))
    } else {
        Ok(Checkout::Rejected("limit exceeded"))
    }
}

fn describe(result: Result<Checkout, String>) -> String {
    match result {
        Result::Ok(Checkout::Paid(product)) => product.label(),
        Result::Ok(Checkout::Rejected(reason)) => reason,
        Result::Err(error) => error,
    }
}

let products = [
    Product::new("book", 4500),
    Product::new("keyboard", 12000),
];
let prices: HashMap<String, i64> = #{
    "book": 4500,
    "keyboard": 12000,
};
let total = products
    .iter()
    .map(|product| product.price)
    .fold(0, |sum, price| sum + price);
let result = checkout(Product::new("book", prices.get("book").expect("book price")), 2);

print("{}; catalog total={}", describe(result), total);
```

Running `ruac build main.rua` produces the following Lua 5.5 source; this is the current compiler's actual output:

```lua
-- Generated by ruac (Rua -> Lua 5.5). Do not edit by hand.
local rua_std = require("rua_std")
assert(rua_std.ABI_VERSION == 2, "incompatible rua_std ABI")
local fmt = rua_std.fmt
local map = rua_std.map
local result__runtime = rua_std.result
local vec = rua_std.vec
local tbcreate = table.create

---@class Product
---@field sku string
---@field price integer
local Product = tbcreate(0, 2)
---@class Checkout

---@return table
local function positive_quantity(quantity)
    if quantity > 0 then
        return result__runtime.ok(quantity)
    else
        return result__runtime.err("quantity must be positive")
    end
end

---@return table
local function checkout(product, quantity)
    local __t1 = positive_quantity(quantity)
    if not __t1[1] then return __t1 end
    local count = __t1[2]
    local total = product.price * count
    if total <= 10000 then
        return result__runtime.ok({ tag = "Paid", product })
    else
        return result__runtime.ok({ tag = "Rejected", "limit exceeded" })
    end
end

---@return string
local function describe(result)
    local __t2 = result[1]
    if __t2 and result[2].tag == "Paid" then
        local product = result[2][1]
        return Product.label(product)
    elseif __t2 and result[2].tag == "Rejected" then
        local reason = result[2][1]
        return reason
    else
        local error = result[2]
        return error
    end
end

---@return Product
function Product.new(sku, price)
    return {
        sku = sku,
        price = price,
    }
end

---@return string
function Product:label()
    return fmt.format("{}: {}", self.sku, self.price)
end

local products = vec.from_table({ Product.new("book", 4500), Product.new("keyboard", 12000), n = 2 })
---@type table
local prices = map.from_table({ ["book"] = 4500, ["keyboard"] = 12000 })
local total = 0
for __t5 = 1, products.n do
    local __t4 = products[__t5]
    __t4 = __t4.price
    total = total + __t4
end
local __t6 = prices:get("book")
if __t6 == nil then error("book price", 2) end
local result = checkout(Product.new("book", __t6), 2)

fmt.print("{}; catalog total={}", describe(result), total)
```

Key semantic mappings item by item:

| Rua | Generated Lua |
|---|---|
| `struct Product` | EmmyLua `Product` class with Lua table |
| `Product::new(...)` | `Product.new(...)` |
| `product.label()` | `Product.label(product)`; method defined with `Product:label()` |
| `Ok(value)` / `Err(error)` | `result__runtime.ok(value)` / `result__runtime.err(error)` |
| `positive_quantity(...) ?` | Checks `[1]` tag, early-returns tagged Result on error |
| `Checkout::Paid(product)` | `{ tag = "Paid", product }` |
| `[a, b]` | 1-based table with explicit `n`, wrapped by `vec.from_table` |
| `#{ key: value }` | `map.from_table({ [key] = value, ... })` |
| `map(...).fold(...)` | A direct accumulating Lua `for` loop |
| `expect("message")` | Nil guard with caller-provided message |

Execution output:

```text
book: 4500; catalog total=16500
```

| Category | Supported syntax |

### Bindings, functions, and expressions

Semicolons terminate statements; blocks, `if`, `match`, and `loop` produce values via tail expressions:

```rua
fn clamp(value: i64, min: i64, max: i64) -> i64 {
    if value < min {
        return min;
    }
    if value > max { max } else { value }
}

let inferred = 42;
let typed: f64 = 3.14;
let mut count = 0;
count += 1;

let computed = {
    let base = 20;
    base + 22
};
```

### Control flow and patterns

```rua
fn bucket(value: i64) -> String {
    match value {
        0 => "zero",
        1 | 2 => "small",
        3..=9 => "medium",
        n if n >= 10 => "large",
        _ => "negative",
    }
}

fn search(values: Vec<i64>, wanted: i64) -> Option<i64> {
    for index in 1..=values.len() {
        if values[index] == wanted {
            return Some(index);
        }
    }
    None
}

let mut stack = [1, 2, 3];
while let Some(value) = stack.pop() {
    if value == 2 { continue; }
    print("{}", value);
}

fn ready() -> bool { true }

let answer = loop {
    if ready() { break 42; }
};
```

`loop` allows `break value`; all break values within the same loop must be type-compatible. `while` and `for` are statements that only accept bare `break;`.

### Struct, enum, trait, and generics

```rua
pub struct Point<T> {
    pub x: T,
    pub y: T,
}

pub enum Shape {
    Dot,
    Circle(f64),
    Rect { width: f64, height: f64 },
}

trait Named {
    fn name(&self) -> String;

    fn display(&self) -> String {
        format("[{}]", self.name())
    }
}

impl Named for Shape {
    fn name(&self) -> String {
        match self {
            Shape::Dot => "dot",
            Shape::Circle(_) => "circle",
            Shape::Rect { .. } => "rect",
        }
    }
}

fn label<T: Named>(value: T) -> String {
    value.display()
}

fn same<T>(value: T) -> T
where
    T: Named,
{
    value
}
```

Inherent `impl Type` defines associated functions and methods; `impl Trait for Type` verifies the trait contract. Operator traits such as `Add`, `Sub`, `Mul`, `Div`, and `Rem` can define binary operator behaviour for user types.

### Option, Result, and error propagation

```rua
fn load(path: String) -> Result<String, String> {
    if path == "" { Err("empty path") } else { Ok("config") }
}

fn load_length(path: String) -> Result<i64, String> {
    let text = load(path)?;
    Ok(text.len())
}

struct Profile {
    city: String,
}

let profile: Option<Profile> = Some(Profile { city: "Shanghai" });
let city = profile?.city ?? "unknown";
let required = profile.expect("profile is required");

let optional_value: Option<i64> = Some(42);
if let Some(value) = optional_value {
    print("{}", value);
}
```

`Option<T>` and `Result<T, E>` provide `map`, `unwrap`, `expect`, `unwrap_or`, and state queries; `?` preserves the current function's Option/Result propagation semantics.

### Vec, HashMap, closures, and iterators

```rua
let mut values = [1, 2, 3];
values.push(4);
let first = values[1];
let first_again = values.first().unwrap();
let last = values.last().unwrap();

let scores: HashMap<String, i64> = #{
    "alice": 10,
    "bob": 20,
};
let known = "alice" in scores;
let contains = 2 in values;
let substring = "ua" in "Rua";

let offset = 10;
let add_offset = |value: i64| -> i64 { value + offset };
let shifted = add_offset(first);
let selected = values
    .iter()
    .map(|value| value + offset)
    .filter(|value| value > 11)
    .collect::<Vec<i64>>();
let total = selected.iter().fold(0, |sum, value| sum + value);
```

`Vec::first()` and `Vec::last()` return `Option<T>` and yield `None` for an empty Vec. Iterators support range/Vec sources, `map`, `filter`, `filter_map`, `find`, `any`, `all`, `count`, `fold`, `collect`, and other adapter/consumer methods; provably one-shot chains generate fused loops directly.

### Modules, extern, attributes, and annotations

Module identity comes from file paths; source code imports definitions via `use`:

```rua
use math::add;
use math::trig::sin as sine;
use text::{join, split as words};

extern "lua" {
    fn host_log(message: String);
    fn host_format(template: String, ...) -> String;
}

#[cfg(feature = "http")]
pub fn serve() {}

#[targets(function)]
#[retention(runtime)]
pub annotation Route(method: String, path: String);

#[Route(method = "GET", path = "/health")]
pub fn health() -> String { "ok" }
```

`extern "lua"` declares ambient Lua globals without generating implementations. For attribute project configuration, targets, retention, metadata, and runtime registry, see [Attributes and conditional compilation](docs/rua-attributes.md) and [Annotations](docs/rua-annotations.md).

## Runtime semantics

A function named `main` is just an ordinary function and is not automatically invoked by the compiler; top-level statements execute in source order.

`Result<T, E>` is a first-class tagged value, not Lua multi-return. `Ok(value)` and `Err(error)` preserve their state through assignment, argument passing, fields, containers, and closure captures; `Ok(nil)` and `Err(nil)` are also distinguishable. `?`, pattern matching, and ordinary return all use the same representation.

```text
Ok(v)  -> { true,  v }
Err(e) -> { false, e }
```

The first array slot is `is_ok`, the second is the payload. The boolean tag is separate from the payload, so `Ok(nil)` and `Err(nil)` remain distinguishable while avoiding named hash-field allocation for every Result.

Only explicit extern/FFI adapters can convert tagged Results to host conventions. `Option<T>` currently uses `T | nil`, so `Some(nil)` is not an expressible state. `Option` and `Result`'s `map`, `unwrap`, `expect`, `unwrap_or`, and state queries come from the standard `.ruai`; `expect(message)` terminates execution with a caller-provided message on a missing value or error state.

```rua
fn load(path: String) -> Result<String, String> {
    if path == "" { Err("empty path") } else { Ok("config") }
}

fn read_config() -> Result<String, String> {
    let config = load("app.rua")?;
    Ok(config)
}
```

Common script expressions remain statically typed and produce explicit Lua control flow:

```rua
struct Profile {
    city: String,
}

fn choose_city(profile: Option<Profile>, cities: Vec<String>) -> String {
    let mut attempts = 0;
    attempts += 1;

    let city = profile?.city ?? "unknown";
    let aliases: HashMap<String, String> = #{ "sh": "Shanghai" };
    let supported = city in cities && "sh" in aliases;

    loop {
        if supported { break city; }
        break "fallback";
    }
}
```

Compound-assignment complex left-hand sides are evaluated only once; `??` only evaluates the right-hand side on `None`, so it won't treat `Some(false)` as a default; `?.` method arguments are also lazily evaluated. `in` supports `Vec`, `HashMap` keys, `String` substrings, and `Iter`/range. `#{...}` constructs a `HashMap` with uniform key-value types. `loop` can return a value via `break value`; `while`/`for` only allow bare `break`.

Other core mappings:

| Rua | Lua runtime representation |
|---|---|
| `i64` / `f64` | Lua integer / number |
| `bool` | boolean |
| `String` | string |
| `Option<T>` | `T` / `nil` |
| `Result<T, E>` | `{ is_ok, payload }` tagged array table |
| `Vec<T>` | 1-based table with `n` length |
| `HashMap<K, V>` | runtime map table |
| `struct` | table + class metatable |
| `enum` | tagged table |

`[1, 2, 3]` is a `Vec<i64>` literal; empty `[]` infers its element type from the expected type. Formatting, output, panics, and assertions use ordinary functions from standard `.ruai` declarations, such as `format("{}", value)`, `print("{}", value)`, and `assert(condition)`, so the compiler and LSP share the same signatures, documentation, and definitions. The Rua expression grammar includes ordinary function calls and Vec literals; function calls uniformly use `callee(...)`, while Vec literals use a separate expression node.

Declarations and members can use `#[cfg(...)]` and `#[cfg_attr(...)]` for project-level conditional compilation. `annotation` declarations provide structured metadata with target, parameter, and retention validation; build metadata is available for external tool scanning, and runtime retention only generates registries when actually used. For complete boundaries and examples see [Attributes and conditional compilation](docs/rua-attributes.md) and [Annotations](docs/rua-annotations.md).

## Modules

Rua's module identity comes directly from file paths; there is no `mod` item in the source grammar. With `src/main.rua` as the entry point, `src` is the source root:

```rua
use math::add;

let total = add(3, 4);
```

Both `src/math.rua` and `src/math/mod.rua` map to `math`, but they cannot coexist; `src/domain/order.rua` maps to `domain::order`, with intermediate directories automatically forming namespaces. `.ruai` files follow the same rules. Workspace `.rua` files take priority over same-path declarations and external libraries; duplicate mappings at the same priority level report ambiguity rather than silently selecting one.

For example, `src/domain/product.rua`:

```rua
pub struct Product {
    pub sku: String,
    pub price: i64,
}

impl Product {
    pub fn new(sku: String, price: i64) -> Product {
        Product { sku, price }
    }

    pub fn label(&self) -> String {
        format("{}: {}", self.sku, self.price)
    }
}
```

Entry point `src/main.rua`:

```rua
use domain::product::Product;

let product = Product::new("book", 4500);
print("{}", product.label());
```

After running `ruac build src/main.rua --emit modules --out-dir dist`, the entry point `dist/main.lua` uses ordinary Lua require:

```lua
-- Generated by ruac (Rua -> Lua 5.5 modules). Do not edit by hand.

local rua_std = require("rua_std")
assert(rua_std.ABI_VERSION == 2, "incompatible rua_std ABI")

local DomainProduct = require("domain.product")

local fmt = rua_std.fmt

---@class Main
local Main = {}

local product = DomainProduct.new("book", 4500)
fmt.print("{}", DomainProduct.label(product))

return Main
```

`dist/domain/product.lua`, with only one public type, returns `Product` directly without adding a `Product.Product` level of indirection:

```lua
-- Generated by ruac (Rua -> Lua 5.5 modules). Do not edit by hand.

local rua_std = require("rua_std")
assert(rua_std.ABI_VERSION == 2, "incompatible rua_std ABI")

local fmt = rua_std.fmt
local tbcreate = table.create

---@class domain.Product
---@field sku string
---@field price integer
local Product = tbcreate(0, 3)
Product.__index = Product

---@return domain.Product
function Product.new(sku, price)
    return setmetatable({ sku = sku, price = price }, Product)
end

---@return string
function Product:label()
    return fmt.format("{}: {}", self.sku, self.price)
end

return Product
```

`.ruai` files allow empty `{}` as function/method declaration bodies, but any non-empty implementation or top-level executable statement is rejected with `E0108`. The compiler and LSP share `rua-project` path mappings; the resolution phase assigns stable identities to modules, definitions, locals, types, traits, variants, and use sites, and Lua codegen only consumes these results.

## Compiler

CLI:

```bash
ruac build src/main.rua                         # writes src/main.lua
ruac build src/main.rua -o dist/app.lua
ruac build src/main.rua --emit modules --out-dir dist \
  --lua-path dist --lua-path /path/to/rua_std
ruac check src/main.rua
ruac build src/main.rua --std-path ./path/to/standard-library
ruac build src/main.rua -c ./path/to/.ruarc.toml
ruac build src/main.rua --features http,trace --cfg runtime=moon
ruac metadata src/main.rua > dist/annotations.toml
```

Default `--emit bundle` writes the entire program into a single Lua file. `--emit modules` generates one file for each runtime module with a concrete `.rua` source: the entry point keeps the root filename, `domain::order` writes to `domain/order.lua`. Each output uses ordinary Lua `require("domain.order")` at the top to load dependencies, with dependency locals using PascalCase of the full logical path, e.g. `application.checkout` becomes `ApplicationCheckout`. Modules with a single public `struct`/`enum` return that type table directly, e.g. `domain/product.rua` returns `Product` so callers use `product.new()` directly; modules with multiple public types or no public types return a module table. Module tables use PascalCase of the filename's last segment, e.g. `presentation/console.rua` produces `local Console = {}`. EmmyLua identities use the full path, e.g. `---@class presentation.Console` or `---@class domain.Product`, to avoid collisions between same-named definitions in different directories. Generated file headers are grouped by Lua search path, `rua_std` ABI, module dependencies, and standard library local aliases; stable blank lines separate types, functions, and the final `return`. Repeatable `--lua-path <dir>` writes search directories into the entry point's `package.path`:

Directory-only namespaces only participate in compile-time path resolution; they do not generate aggregation files like `domain.lua` or produce extra requires. If `require("domain")` is needed, a concrete `domain.rua` must be provided.

```bash
lua dist/main.lua
```

The runtime environment can also be configured via `LUA_PATH` or persisted in `.ruarc.toml` under `runtime.lua_path`. Modules mode follows Lua `require`'s depth-first initialization order; circular module dependencies that cannot be safely represented with ordinary `require` are reported as compile-time errors—such projects can eliminate the cycle or continue using bundle mode. `--out-dir` must be explicitly specified; the compiler overwrites files it generates for the current build but does not delete other content in the directory.

When runtime annotations are present, the modules artifact includes an aggregated `rua_annotations.lua`. Loading the registry only registers the canonical locator; `Annotations::load` lazily loads targets by module/path.

`ruac` by default searches upward from the input file directory for the nearest `.ruarc.toml`. Project configuration is shared with the LSP and uses snake_case fields:

```toml
[workspace]
library = ["./types", "../host/moon.ruai"]

[[workspace.lua_library]]
root = "../moon_rs/lualib"

[workspace.library_mounts]
host = "../host/actual-name.ruai"

[runtime]
std_path = "./std"
lua_path = ["./dist", "/opt/rua/lib"]

[build]
features = ["http", "trace"]

[build.cfg]
runtime = "moon"
embedded = false
```

Ordinary Lua libraries should prefer `workspace.lua_library`. Use `root` for co-located directories; when declarations and runtime are separate, write a single root pair:

```toml
[[workspace.lua_library]]
declaration_root = "../moon_rs/ruai"
runtime_root = "../moon_rs/lualib"
```

Declarations are automatically mapped by relative path: `moon/http/client.ruai` corresponds to Rua `moon::http::client` and Lua `require("moon.http.client")`. Each file-based `.ruai` is an independent Lua module. Both the compiler and LSP recursively index declaration roots; runtime roots are automatically added to generated entry points' `package.path`, so no per-module configuration or source declarations are needed.

`workspace.library` configures declaration-only search paths; `library_mounts` configures explicit logical name mappings. Command-line `--library`, `--library-mount name=path`, `--std-path`, and repeatable `--lua-path dir` can override or supplement project configuration. `lua_path` sets Lua runtime search directories for bundle/modules generated entry points and does not participate in Rua/`.ruai` compile-time module resolution.

Library API:

```rust
let lua = ruac::compile_str("let value = 42;")?;
let lua = ruac::compile_path(path)?;
let lua = ruac::compile_path_with_std(path, std_root)?;
let lua = ruac::compile_project(&project_spec, &source_provider)?;
let artifact = ruac::compile_project_with_diagnostics(
    &project_spec,
    &source_provider,
)?;
let modules = ruac::compile_project_modules_artifact(&project_spec, &source_provider)?;
```

`compile_project_with_diagnostics` is the full host integration entry point: success values include Lua with a generated-to-Rua source map; failure values (`CompileFailure`) include diagnostic codes, files, and byte ranges. `compile_str`, `compile_path`, `compile_project`, and artifact convenience APIs also return structured failures; only the CLI is responsible for rendering display text. The default entry point uses the embedded standard library; `compile_*_with_std` and `--std-path` explicitly load a directory containing `std.toml`. `compile_path_artifact` preserves the generated-to-Rua source map for filesystem hosts, and `ruac::stacktrace::convert_lua_traceback` converts Lua runtime traceback frames back to Rua ranges; see [Lua stack conversion](docs/rua-stacktrace.md). `ProjectSpec` provides stable `FileId`, logical paths, source roots, and library mounts; `SourceProvider` provides source text.

The standard library's `.ruai` signatures, documentation, members, Lua packages, exported sub-tables, local aliases, and optional ABI are uniformly described by `std.toml`. Declaration files and `rua_std.lua` reside in the same resource directory; `Vec`, `HashMap`, `Iter`, `String`, `Result`, formatting, and integer operations are separate exports of the same Lua package. The compiler only reserves special representations for the `Option` and `Result` language items specified in the manifest. User types with the same names remain ordinary `struct`/`enum` and are not mistakenly lowered through the standard library. Deployment only requires providing a single `rua_std.lua`.

Compiler main pipeline:

```text
rua-lex tokens
  -> strict parser / owned AST
  -> module collection + resolved HIR identities
  -> structural check + ID-keyed type facts
  -> backend layout
  -> structured Lua IR
  -> Lua printer
```

## IDE

Rua uses two parsers:

- `ruac` uses a fail-fast strict parser, suitable for the compiler and embedded hosts.
- `rua-syntax` uses an error-tolerant Rowan parser that preserves trivia and error nodes, suitable for source code under editing.

Both share `rua-lex`, `rua-core`, `rua-project`, grammar corpus, and range conformance tests, but do not share AST, error recovery, or semantic implementation. `rua-analysis` is an independent incremental semantic engine that does not call the compiler AST/typeck as a production fallback.

`rua-lsp` supports hover, goto definition/implementation, completion, references, atomic rename, call/type hierarchy, inlay hints, diagnostics, semantic tokens, code actions, symbols, folding, and formatting. Documentation for functions, methods, trait methods, extern declarations, and `.ruai` declarations comes from a unified semantic record; standard functions such as `print`, `format`, and `panic` are indexed directly from the standard `.ruai`. Enum variants use the same identity in construction, qualified paths, aliases, and patterns.

VS Code settings:

- `rua.server.path`, `rua.server.args`, `rua.compiler.path`,
  `rua.compiler.args`, `rua.trace.server`.

Library, module mount, and standard library paths are written only in the workspace root's `.ruarc.toml`, so that `ruac` and `rua-lsp` use the same project inputs. VS Code settings only control the compiler, language server process, and protocol trace. Right-click a `.rua` file in Explorer or the editor to run **Rua: Build File**.

Workspace/library scanning respects `.gitignore`, `.ignore`, and `.ruaignore`, excluding `.git`, `target`, and `node_modules` by default.
